3-way merge vs 2-way merge

What’s 3-way and 2-way merge? So far, what you’ve seen is a 3-way merge in action in the previous part:

You’ve the original file, and you’ve modified this file in the same place in two different branches. The result file is the one created after the changes are combined, and you decided what is the final code.

2-way merge is easier to explain. It doesn’t consider the base file also known as common ancestor, to calculate the merge.

Let’s go back to the “dokan-net-0.6.0\DokanNet\DokanOperations.cs”, make a couple of very simple changes, and try to merge with a 2-way merge tool. Check the results on the below figure:

At each difference the 2-way merge tool won’t be able to decide whether it is modified or removed code. A 3-way merge tool would automatically solve the same two conflicts. Of course, Plastic SCM only uses 3-way merge tool.
